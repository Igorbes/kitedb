## KiteDB = Spring JDBC + Velocity Template + Simple Flat Mapper (SFM)

This framework is an additional abstraction layer for working with databases using [Spring JDBC](https://spring.io/guides/gs/relational-data-access), [Velocity Templates](https://velocity.apache.org), and [SFM](https://simpleflatmapper.org/).

## About The Project

Here's why:
### Main advantages over JPA
#### 1. Full control over SQL
Unlike JPA, where the final SQL is often generated by the ORM and may be non-obvious, here SQL is written explicitly in Velocity templates. For example, the user controls select, join, where, copy, and count in select.sql.vm.

##### This provides the following advantages:
* it is easier to optimize queries for a specific database;
* it is easier to use PostgreSQL-specific features;
* it is easier to analyze the query plan;
* there is no unexpected SQL that an ORM might generate;
* there is less risk of getting inefficient fetch strategies.
#### 2. No heavyweight persistence context
JPA usually works through a persistence context, dirty checking, lazy loading, entity lifecycle, and proxy objects. In KiteDB, the model is simpler: work is done with regular POJOs.

Advantage: less hidden behavior. An object is just an object, not a managed entity with a complex lifecycle.

#### 3. No N+1 problem in the classic JPA sense
In JPA, N+1 often occurs because of lazy relationships and implicit additional loading. In KiteDB, relationships are explicitly defined through a data graph and an SQL template. For example, a join for sales is included only if the field is attached in the graph: select.sql.vm.

That is, the developer can see in advance which relationships will be loaded and can control this through DataGraphBuilder.graph(), DataGraphBuilder.required(), and DataGraphBuilder.joinTemplate().

#### 4. Flexible selection of only the required fields
A very strong idea of the project is DataGraphBuilder. The graph makes it possible to specify which fields are needed for a specific operation: DataGraphBuilder.required(), DataGraphBuilder.key(), DataGraphBuilder.delete(), DataGraphBuilder.intercept().

Compared with JPA, this is similar to dynamic projections/entity graphs, but simpler and closer to SQL. You can create one base graph and then, at runtime, remove unnecessary fields or keep only the required ones. This is useful for APIs where different endpoints require different data sets.

#### 5. Explicit and predictable result mapping
KiteJdbcTemplateImpl.select() uses SimpleFlatMapper: aliases and keys are added, and unordered join mapping is enabled. Graph keys are passed to the mapper through objectGraph.keys(), and aliases through objectGraph.aliases().

This provides convenient mapping of complex results without JPA entity state management.

#### 6. Well suited for read-heavy scenarios
If an application reads a lot of data, especially with dynamic field sets and joins, KiteDB looks more advantageous than JPA. SQL remains manual and optimizable, while object assembly is automated through SFM.

In such scenarios, JPA often requires:

* DTO projections;
* custom queries;
* entity graphs;
* fetch joins;
* manual lazy/eager control;
* fighting duplicates in one-to-many joins.

In KiteDB, this model is built directly through the graph and SQL templates.

#### 7. Simple DAO interfaces without manual implementation
A DAO is created through a proxy in KiteDB.create(). The insert, update, delete, select, count, copy, execute, and graph methods are routed inside the invocation handler: KiteDB.java.

This is similar to the convenience of a Spring Data JPA repository, but without ORM and without generating SQL by naming convention. SQL remains in templates, and the interface remains compact.

#### 8. Unified facade for CRUD + count + copy + execute
DaoFullAccess combines selection, insertion, update, deletion, counting, copy, graph, and execute. This provides a convenient high-level abstraction over JDBC.

Compared with JPA, this is simpler and closer to real SQL operations. Compared with plain Spring JDBC, it has less boilerplate code.

#### 9. Batch insert/update/delete and RETURNING support
KiteJdbcTemplateImpl.run() shows that insert/update/delete go through a shared mechanism for batch operations, PreparedStatement, and generated keys. Returned values are handled through Returning, which is used in KiteJdbcTemplate.

This is stronger than a plain JdbcTemplate in terms of ready-made infrastructure: the user does not need to write repetitive batch code and generated-key handling every time.

#### 10. PostgreSQL COPY support
A separate advantage over JPA is the copy operation. In KiteJdbcTemplateImpl.copy(), SQL is built through a template and then passed to CopyManager.

JPA is poorly suited for such tasks, and a plain JdbcTemplate would require a manual low-level implementation.

### Main advantages over Spring JdbcTemplate
#### 1. Less manual boilerplate code
   With plain Spring JdbcTemplate, you usually need to manually write:

* SQL;
* parameters;
* RowMapper or ResultSetExtractor;
* batch setter;
* generated-key handling;
* count queries;
* copy/export logic;
* DAO implementations.

In KiteDB, a significant part of this is moved into a common layer: KiteJdbcTemplate, KiteDB, and DaoFullAccess.

#### 2. SQL templating through Velocity
A plain JdbcTemplate usually works with SQL strings. Here SQL can be built dynamically through Velocity and the $k context.

This is more convenient for complex queries where different operations use one base template:

* select;
* count;
* copy;
* optional fields;
* optional joins;
* conditional filters.

In select.sql.vm, one template switches between copy, count, and regular select.

#### 3. Automatic mapping of complex join results
Spring JdbcTemplate by itself does not conveniently solve one-to-many mapping. Usually, you have to write a ResultSetExtractor manually: a map by id, collection aggregation, and deduplication.

In KiteDB, this is handled by the combination of a graph, keys, aliases, and SFM. In KiteJdbcTemplateImpl.select(), a mapper is created, aliases and keys are passed to it, and then the result is assembled through ResultSetExtractor.

#### 4. DataGraph as a declarative layer over SQL
Spring JdbcTemplate has no concept of a "data graph". In KiteDB, the graph becomes a contract between Java code and the SQL template.

For example, DataGraphBuilder.required() says that a field is needed, and $k.optional in the SQL template decides whether to include the column in SQL.

This makes data selection controllable at the Java API level without giving up SQL.

#### 5. DAO proxy over JDBC
Spring JdbcTemplate does not create DAO interfaces by itself. In KiteDB, a DAO is created through KiteDB.create(), and operations are linked to templates through annotations, as shown in TestKiteDao.java.

This provides the convenience of the repository approach without switching to JPA.

#### 6. Reusing one SQL template for different operations
In select.sql.vm, the same view block is used for regular select, count, and copy. This reduces SQL duplication.

With plain JdbcTemplate, you would often need separate SQL strings or separate methods for each mode.

#### 7. Extensibility through lower levels
The project has several abstraction levels: the DAO level, KiteJdbcTemplate, and the underlying Spring NamedParameterJdbcTemplate through KiteJdbcTemplate.getJdbcTemplate(). If the higher level is not enough, you can go lower.

You can also extend a DAO through DaoFacade, as shown in TestKiteDao.java.

## Core

### Abstraction levels
You can use KiteDB in 3 levels of abstraction.
#### Major interfaces (Top-level).
You should implement interfaces to execute DB operations over User objects.
```
1. DaoInsert<User, Integer>.class (Insert object)
2. DaoSelect<User, UserFilter>.class (Filter objects)
3. DaoUpdate<User, Integer>.class (Update objects)
4. DaoDelete<Integer>.class (Delete objects)
5. DaoCount<User, UserFilter>.class (Calculate count objects)
6. DaoCopyable<User, UserFilter>.class (Copy objects to output stream)
7. Executable (Execute complicated operations in a transactional manner)
```
For convenience, you can use [DaoFullAccess.java](src/main/java/pro/kitedb/dao/DaoFullAccess.java)
```
public interface DaoFullAccess<Ident, Obj, F extends Filter<? super Obj>> extends
        DaoSelect<Obj, F>,
        DaoDelete<Ident>,
        DaoUpdate<Obj, Ident>,
        DaoInsert<Obj, Ident>,
        DaoCount<Obj, F>,
        DaoCopyable<Obj, F>,
        Graphable<Obj, F>,
        Executable {
}
```
UserDao interface declaration for operations over a User object.
```java
@DaoInsertAbility(template = "sql/user/insert.sql.vm", clazz = User.class, ppss = UserInsertPPSS.class)
@DaoSelectAbility(template = "sql/user/select.sql.vm", graphable = UserGraph.class)
@DaoUpdateAbility(template = "sql/user/upsert.sql.vm", ppss = UserUpsertPPSS.class)
@DaoDeleteAbility(template = "sql/user/delete.sql.vm")
@DaoCopyAbility(template = "sql/user/select.sql.vm")
public interface UserDao extends DaoFullAccess<Integer, User, UserFilter> {}
```
Creating a UserDao instance.
```java
UserDao userDao = new KiteDB(dataSource).create(UserDao.class);
```
To extend the functionality of the DAO layer, you can create your own class and inherit from [DaoFacade.java](src/main/java/pro/kitedb/dao/DaoFacade.java). You can add missing methods in the new class.
```java
class UserExtendDao extends DaoFacade<Number, User, UserFilter> {
    private final @Getter UserDao implementation;
    private final @Getter KiteJdbcTemplate jdbcTemplate;

    public UserExtendDao(KiteJdbcTemplate kiteJdbcTemplate) {
        this.jdbcTemplate = kiteJdbcTemplate;
        this.implementation = new KiteDB(kiteJdbcTemplate.getJdbcTemplate().getJdbcTemplate().getDataSource()).create(UserDao.class);
    }

    public long myMethod() throws DataException {
        return this.jdbcTemplate.count("sql/user/select.sql.vm", new UserGraph(jdbcTemplate).graph(UserFilter.builder().build()));
    }
}
```

#### [KiteJdbcTemplate.java](src/main/java/pro/kitedb/KiteJdbcTemplate.java) (Second-level).
```
public interface KiteJdbcTemplate extends
        InsertJdbcTemplate,
        SelectJdbcTemplate,
        UpdateJdbcTemplate,
        DeleteJdbcTemplate,
        CountJdbcTemplate,
        CopyJdbcTemplate,
        ExecuteJdbcTemplate {
    KiteNamedParameterJdbcTemplate getNamedParameterJdbcTemplate();
}
```
#### [KiteNamedParameterJdbcTemplate.java](src/main/java/pro/kitedb/KiteNamedParameterJdbcTemplate.java) (Third-level).
Extends org.springframework.jdbc.core.namedparam.NamedParameterJdbcTemplate when the methods at higher abstraction levels are not enough.



### POJO
You work with POJO and enjoy its simplicity and convenience. You can put it in sessions, serialize it, and use it in other operations.
```java
@NoArgsConstructor
@AllArgsConstructor
public class User {
    private @Getter @Setter Integer id;
    private @Getter @Setter String name;
    private @Getter @Setter String surname;
    private @Getter @Setter Collection<Sale> sales;
}

@NoArgsConstructor
@AllArgsConstructor
public class Sale {
    private @Getter @Setter Integer id;
    private @Getter @Setter Integer userId;
    private @Getter @Setter LocalDate timestamp;
    private @Getter @Setter Integer amount;
    private @Getter @Setter Integer summary;
    private @Getter @Setter Address address;
}
```

### Filters

### DataGraph
An object graph is a set of fields fetched from the DB and join constructs. It can be created through a constructor, but the optimal option is to use [DataGraphBuilder.java](src/main/java/pro/kitedb/graph/DataGraphBuilder.java).
```java
public DataGraph<User, User.UserFilter> graph(User.UserFilter filter) throws DataException {
    return DataGraphBuilder.<User, UserFilter>graph(User.class, filter)
            .key("id")
            .required("name")
            .required("surname")
            .required("sales", (userGraph) -> new SaleGraph(this.jdbcTemplate).graph(filter.getSaleFilter()))
            .build();
}

public DataGraph<Sale, SaleFilter> graph(SaleFilter filter) throws DataException {
    return DataGraphBuilder.<Sale, SaleFilter>graph(Sale.class, filter)
            .key("id")
            .required("userId")
            .required("timestamp")
            .required("amount")
            .required("summary")
            .required(new AsteriskFieldGraphReader(Address.class, "address"))
            .joinTemplate(new JoinTemplateImpl("sql/sale/select.sql.vm", jdbcTemplate))
            .build();
}
```
#### key
Sets the key for the graph. There may be several keys if the primary key is composite. Rows are collapsed by key to obtain unique objects (see SimpleFlatMapper).
```
public DataGraph<Sale, SaleFilter> graph(SaleFilter filter) throws DataException {
    return DataGraphBuilder.<Sale, SaleFilter>graph(Sale.class, filter)
            .key("id")
            .key("userId")
            .build();
}
```
#### required
Declares which field must be fetched for the current operation. For example: id, name, surname, and sales.
```java
public DataGraph<User, User.UserFilter> graph(User.UserFilter filter) throws DataException {
    return DataGraphBuilder.<User, UserFilter>graph(User.class, filter)
            .key("id")
            .required("name")
            .required("surname")
            .required("sales", (userGraph) -> new SaleGraph(this.jdbcTemplate).graph(filter.getSaleFilter()))
            .build();
}
```
dataGraph can be changed at runtime. For example, you can use a base field set in the default graph and add or remove fields when necessary.
```java
public DataGraph<User, User.UserFilter> graph(User.UserFilter filter) throws DataException {
    return DataGraphBuilder.<User, UserFilter>graph(User.class, filter)
            .key("id")
            .required("name")
            .required("surname")
            .required("sales", (userGraph) -> new SaleGraph(this.jdbcTemplate).graph(filter.getSaleFilter()))
            .build();
}
/* exclude sales from the selection */
public DataGraph<User, UserFilter> graphWithoutSales(UserFilter filter) throws DataException {
    return graph(filter)
            .toBuilder()
            .delete("sales")
            .build();
}
/* another option is to keep only the required fields */
public DataGraph<User, UserFilter> graphWithoutSales2(UserFilter filter) throws DataException {
    return graph(filter)
            .toBuilder()
            .intercept("id", "name", "surname")
            .build();
}
```
#### PlainObjectGraph
Several fields from one table can be combined into one class. For example, postal code, address, and city can be combined into a Sale.Address.class object.
```java
public DataGraph<Sale, SaleFilter> graph(SaleFilter filter) throws DataException {
        return DataGraphBuilder.<Sale, SaleFilter>graph(Sale.class, filter)
                .key("id")
                .required(new AsteriskFieldGraphReader(Sale.Address.class, "address"))
                .build();
    }
```
#### Join template
A class that participates as a parent object field through join operations requires an sql.vm template. JoinTemplate is optional if the object is not used as a join in other objects. Usually this is a select.sql.vm template, but in some cases it may be another template.
```java
public DataGraph<Sale, SaleFilter> graph(SaleFilter filter) throws DataException {
        return DataGraphBuilder.<Sale, SaleFilter>graph(Sale.class, filter)
                .key("id")
                .joinTemplate(new JoinTemplateImpl("sql/sale/select.sql.vm", jdbcTemplate))
                .build();
    }
```
### SQL Context
Database queries are written in templates processed by Apache Velocity (see the [Apache Velocity](https://velocity.apache.org/) documentation). The optimal option is to use the .sql.vm file extension, so the editor can still correctly detect both the SQL language structure and the template structure. [KiteContext.java](src/main/java/pro/kitedb/context/KiteContext.java) is passed to the template as the $k variable. This context is used to build the final SQL script depending on the provided graph and filter. Templates are searched in classpath resources (see the [Apache Velocity](https://velocity.apache.org/) documentation). You can override the template lookup mechanism by overriding the KiteJdbcTemplate.init method.

/sql/user/select.sql.vm
```sql
#define($view_main)
    select
        $k.optional('u.id')
        $k.optional('u.name')
        $k.optional('u.surname')
        $k.optional('u.nickname')
        #if(!$k.isCount())
           $k.optional('s.sales')
        #end
        $k.end()
    from public.User u
    #if($k.isJoined('sales') && !$k.isCount())
        #set($sales = $k.join('sales'))
        left join ($sales.sql()) s on s.$sales.joinKey('userId') = u.id
    #end
    where true

    #set($filter = $k.filter())
    #if($filter.present('id'))
      and u.id = $filter.param('id')
    #end
    #if($filter.present('without_nickname'))
      and u.nickname is null
    #end
#end

#if($k.isCopy())
    copy ($view_main) to stdout with (format csv, header)
#elseif($k.isCount())
    select count(1) from ($view_main) v
#else
    select v.* from ($view_main) v
#end
```

/sql/sale/select.sql.vm
```sql
select
    $k.optional('s.id')
    $k.optional('s.user_id', 'userId')
    $k.optional('s.timestamp')
    $k.optional('s.amount')
    $k.optional('s.summary')
    $k.optional('s.postal', 'address.postal')
    $k.optional('s.city', 'address.city')
    $k.optional('s.street', 'address.street.name')
    $k.end()
from public.Sale s
where true
#set($filter = $k.filter())
#if($filter.present('amount'))
    and s.amount = $filter.param('amount')
#end
```
```sql
$k.optional - returns the SQL construct 'u.name as "User.name",' when .required("name") is set
$k.isJoined('sales') - returns true if the sales DataGraph was added to DataGraph
$k.end() - finishes the selection with an 'end' column at the end so the SQL construct remains valid
$k.join('sales') - returns the sales DataGraph
$sales.sql() - builds and returns the sales SQL
$sales.joinKey('userId') - returns the alias of the Sales.user_id column from the public.Sale table
$k.graph() - returns the graph of the current context
$k.graph().alias('id') - returns the alias of the User.user_id column from the public.User table
$k.isCopy(), $k.isCount(), $k.isSelect(), etc. - define the operation method (CRUD + Execute + Count)
```
Column aliases are formed using [SqlFieldPrefixFactory.java](src/main/java/pro/kitedb/graph/prefix/SqlFieldPrefixFactory.java) + the column name. It can be passed when building a graph through the static methods of [DataGraphBuilder.java](src/main/java/pro/kitedb/graph/DataGraphBuilder.java).
### Filter
Data filtering through [Filter.java](src/main/java/pro/kitedb/graph/Filter.java) is performed through PreparedStatement to prevent injections. The filter is passed to DataGraph when it is created through a method or through the static methods of [DataGraphBuilder.java](src/main/java/pro/kitedb/graph/DataGraphBuilder.java).
```java
$k.filter() - returns the filter
$filter.present('id') - returns true if the corresponding key was passed. The key value can be anything, including null
$filter.get('id') - returns the Prepared parameter ':User.id'
```
### Variable
If the data explicitly excludes any injections, you can use [Variabale.java](src/main/java/pro/kitedb/graph/Variabale.java) to pass it into a query. For example, you can pass Enum.ASC or Enum.DESC, limit, or offset as variables.
```java
update public.User set nickname = $k.var('nickname') where nickname is null;
```
To prevent SQL injection through [Variabale.java](src/main/java/pro/kitedb/graph/Variabale.java), pass the variable explicitly through a Prepared parameter.
```java
update public.User set nickname = :nickname where nickname is null;
```
### Returning
When executing an SQL script, you can return rows and collect values into an object through RETURNING and the [Returning.java](src/main/java/pro/kitedb/factory/returning/Returning.java) interface.

```java
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.transaction.annotation.Transactional;
import pro.kitedb.graph.DataGraph;

@Bean
public KiteJdbcTemplate kiteJdbcTemplate(DataSource dataSource) {
    return new KiteJdbcTemplate(dataSource);
}

    return new KiteDB(dataSource).create(UserDao.class);

@Transactional
@Repository
class UserExtendDao extends DaoFacade<Number, User, UserFilter> {
    private @Autowired @Getter UserDao implementation;
    private @Autowired KiteJdbcTemplate jdbcTemplate;

    public long myMethod(DataGraph<User, Integer> dataGraph) throws DataException {
        return this.jdbcTemplate.count("sql/user/select.sql.vm", dataGraph);
    }
}
```

## Summary
KiteDB does not try to replace JPA in every scenario. If you need a rich ORM model with cascade, dirty checking, entity lifecycle, second-level cache, and JPQL, JPA will be more familiar.

### Where KiteDB is especially stronger.
1. Complex SQL queries where it is important to control every column and join.
2. Read-heavy APIs where different screens require different field sets.
3. PostgreSQL projects that need RETURNING and COPY.
4. Services where JPA is too heavyweight, but plain JdbcTemplate is too verbose.
5. Projects where you need to keep an SQL-first approach while removing manual mapping and DAO boilerplate.
6. Reports, exports, dynamic selections, and complex filters.
7. Batch insert/update/delete with key returns.


However, if the priority is SQL control, performance, dynamic selection composition, a simple POJO model layer, and less boilerplate compared with JdbcTemplate, then KiteDB looks like a successful compromise between JPA and Spring JdbcTemplate.
